# Frontend Design Guidelines

> Last update: 2026-04-20

## 目标

本规范用于统一 `alter0` Web 前端的视觉方向、页面结构、实现约束与验收口径，确保页面在保持稳定工程边界的同时，具备清晰的产品气质与完成度。

## 适用范围

- `internal/interfaces/web/frontend` 下的页面、组件、样式与交互改动
- `internal/interfaces/web/static/dist` 对应的 Web Shell、Chat、Agent、Terminal、Control、Memory、Sessions、Tasks、Products 等浏览器界面
- 对外公开的 Product 页面、独立 HTML 空间和其他品牌展示型 Web 页面

## 产品默认视觉立场

- `alter0` 的默认前端形态是高密度、长时间使用的工作台，而不是营销站模板。
- 稳定页面优先服务执行、阅读、配置、筛选、追踪和输入，不通过大块 hero、装饰口号或卡片拼贴制造存在感。
- 视觉表达需要克制但不平庸：允许明确风格、强层级和细节打磨，但不得牺牲信息密度、可读性、可操作性和响应速度。

## 页面建模

开始实现前，先确定以下四项：

1. 页面类型：工作台页、对话页、详情页、配置页、公开 Product 页或独立宣传页
2. 首屏主任务：阅读、输入、选择、筛选、比较、配置、执行或转化
3. 视觉命题：页面的气质、材质感、信息密度和动效力度
4. 内容结构：首屏、支撑区、细节区、收口动作分别承担什么职责

同一页面的首屏只允许一个主视觉重点。若首屏已经承担品牌表达，就不要再叠加统计条、徽章云、次级入口、平行卖点和额外说明区；若首屏承担工作流入口，就不要再套一层营销式 hero。

## 页面类型规则

### 工作台与控制页

- `Chat / Agent / Terminal / Memory / Sessions / Tasks / Models / Environments / Cron / Codex Accounts / Skills / MCP / Channels / Products Workspace` 默认采用工作台式信息架构。
- 首屏优先暴露当前任务、列表、输入区、状态栏或配置区，不渲染额外 hero、品牌宣言、浮层徽章或装饰性摘要卡。
- 默认使用克制配色、少量强调色、紧凑但可读的排版与稳定的阅读宽度。
- 默认不使用大面积装饰渐变和过多投影，不把页面退化成“卡片墙”。

### 公开 Product 页与独立 HTML 页面

- 允许更强品牌叙事、图像主导和更鲜明的情绪氛围。
- 首屏仍保持单一构图：品牌或产品名、一个主标题、一句支撑文案、一个主 CTA 组和一个主视觉平面。
- 公开页面的视觉锚点应承担叙事职责，不使用缺少语义的抽象拼贴、悬浮小卡片或假 3D 装饰替代真实视觉焦点。

## 视觉与排版规则

### Typography

- 使用有明确气质的字体组合，不以 `Inter`、`Roboto`、`Arial`、系统默认字体作为首选。
- 建立显示级、标题级、正文级、标签级四类基础排版角色，并通过设计令牌统一控制。
- 标题负责建立记忆点，正文负责快速扫读；避免所有文本都落在相近字号和字重上。

### Color and Tokens

- 先定义 `background / surface / text / muted / accent / border / success / warning / danger` 等令牌，再编写组件样式。
- 默认使用克制主色 + 少量强调色，不在单页中堆叠多个竞争性色相。
- 避免“紫色渐变 + 白底 + 毛玻璃卡片”这类无上下文的通用 AI 视觉。

### Composition

- 优先通过对齐、留白、比例、裁切、分栏和视线流向组织页面，不靠堆组件制造结构。
- 默认不把卡片作为基底布局。卡片只在信息边界或交互边界真实存在时使用。
- 每个 section 只做一件事，标题和正文不重复表达同一信息。

### Motion

- 动效用于建立层级、反馈状态、强调切换，不作为持续装饰噪音。
- 优先少量高价值动画，如首屏进入、抽屉展开、步骤切换、状态回填。
- 固定元素、浮层、抽屉、runtime sheet 与底部面板必须始终避开文本、按钮和输入区的核心操作面。

### Copy

- 页面文案使用产品语言，不暴露提示词、实现说明或设计评论。
- 标题负责表达主意图，说明文字控制在可一眼扫完的长度。
- 避免在页头、section 标题、描述文案和 CTA 上重复表达同一意思。

## alter0 壳层规则

- 桌面端保持全宽三栏工作台；`1100px` 及以下切换为抽屉式单列；`760px` 及以下继续压缩按钮与抽屉宽度。
- `Chat` 空态欢迎区、`route-head`、`Terminal` 工作区与控制页正文继续遵守现有贴顶节奏和阅读宽度，不为单页视觉试验破坏稳定壳层。
- `Chat` 空态不额外挂载 workspace hero；页面类路由不额外挂载 route hero；主导航和会话栏不重复页面摘要。
- `Agent` 运行态与 `Terminal` 的阅读焦点优先落在消息、Process、输出区和输入区，不让装饰层抢占首屏。
- 所有布局调整必须兼容 `VisualViewport`、移动端键盘贴底、会话设置底部面板、抽屉开合和滚动定位条。

## 实现流程

### 1. 先定方向

- 明确页面类型、主任务、首屏构图和视觉命题
- 定义主要排版角色、颜色令牌和布局骨架

### 2. 再定结构

- 确认首屏只承载一个主重点
- 为每个 section 指定单一职责
- 明确桌面、平板、手机三档信息裁剪策略

### 3. 最后实现细节

- 复用现有壳层、断点、路由容器与共享样式基线
- 仅在确有必要时新增视觉层、卡片层或浮层
- 对固定元素、浮动操作与动效做跨尺寸安全检查

## 验收清单

- 首屏是否存在唯一主重点，而不是多个模块争抢注意力
- 字体、颜色、间距、边框和阴影是否来自同一套令牌，而不是临时拼接
- 去掉多余边框、阴影和背景后，是否仍能保留清晰层级；若可以，则应继续收敛卡片化处理
- 页面标题、说明和主操作是否各自承担不同职责，没有重复表达
- 桌面、窄屏与真手机宽度下，抽屉、输入区、底部面板和固定元素是否都不遮挡关键内容
- 页面是否延续 `alter0` 作为工作台产品的稳定气质，而不是退化为通用 AI 模板或营销页套壳
