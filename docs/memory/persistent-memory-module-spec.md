# alter0 Memory 模块统一说明与需求文档

> 版本：v2.0
> 状态：Draft / Requirement Baseline
> 适用范围：Chat、Agent、共享 Memory 内核、Memory 页面说明文档视图
> 关联需求：R-017、R-018、R-020、R-024、R-025、R-026、R-027、R-028、R-031、R-032

## 1. 目标

为 `alter0` 建立一套统一、可审计、可回溯、可演进的 Memory 体系，同时满足以下目标：

1. `Chat` 保持多轮对话连续性，并能在跨会话场景下稳定召回偏好、长期事实与历史任务摘要。
2. `Agent` 在执行型任务中复用历史决策、项目规则、长期偏好与任务产物，减少重复探索。
3. 记忆体系坚持 Markdown 主存与可人工维护原则，不依赖不可解释的黑盒状态。
4. 前端 `Agent -> Memory` 提供统一只读查阅入口，覆盖长期记忆、天级记忆、强制上下文、任务历史与本说明文档。

## 2. 范围

本说明文档覆盖以下内容：

1. `system` 上下文文件职责、优先级与加载顺序。
2. 会话历史、会话摘要、天级记忆、长期记忆、任务记忆的分层模型。
3. `Chat` 与 `Agent` 在读取、写入、压缩、召回、可视化上的差异化要求。
4. `.alter0` 运行态目录、任务历史回链与记忆检索接口约束。
5. `Agent -> Memory` 页面展示边界与验收口径。

本说明文档不覆盖以下内容：

1. 向量库、embedding provider、检索服务的最终技术选型。
2. 多租户鉴权与权限模型的最终设计。
3. `Terminal` 模块的专属记忆策略。

## 3. 设计原则

1. Markdown 为主真相源。长期记忆、天级记忆与说明文档统一以 Markdown 持久化。
2. 记忆分层治理。`system`、会话历史、长期记忆、任务历史分层建模，禁止混写。
3. 读写分离。读取链路负责上下文构建，写入链路负责抽取、归档、升级、去重与回链。
4. 默认保守注入。只有在相关性、作用域和预算满足条件时，记忆才进入 prompt。
5. 可审计可回链。所有记忆写入、升级、压缩、召回都必须保留来源引用。
6. 共享内核、策略分流。`Chat` 与 `Agent` 共享底层 Memory 内核，但保留不同产品策略和默认预算。

## 4. 文件职责模型

| 文件 | 角色 | 更新方式 | 生命周期 | 默认是否纳入 Git |
| --- | --- | --- | --- | --- |
| `SOUL.md` | 强制上下文、不可降级规则 | 人工维护 | 长期 | 是 |
| `USER.md` | 用户画像、协作偏好、输出偏好 | 人工维护 | 长期 | 是 |
| `.alter0/agents/<agent_id>/AGENTS.md` | 当前 Agent 私有的执行规范、协作规则与补充约束 | 人工维护 | 长期 | 否 |
| `.alter0/memory/long-term/MEMORY.md` | 长期记忆主存 | 系统升级写入、人工维护 | 长期 | 否 |
| `.alter0/memory/YYYY-MM-DD.md` | 天级记忆日志 | 系统追加写入 | 天级滚动 | 否 |
| `.alter0/tasks/{task_id}/meta.json` | 任务状态与基础元数据 | 系统写入 | 任务生命周期 | 否 |
| `.alter0/tasks/{task_id}/logs.jsonl` | 任务阶段日志 | 系统追加写入 | 按留存策略 | 否 |
| `.alter0/tasks/{task_id}/artifacts.json` | 任务产物索引 | 系统写入 | 按留存策略 | 否 |
| `docs/memory/persistent-memory-module-spec.md` | Memory 主说明文档 | 仓库维护 | 长期 | 是 |

## 5. 记忆分层模型

| 层级 | 名称 | 作用 | 主载体 | 默认写入方式 | 默认读取方式 |
| --- | --- | --- | --- | --- | --- |
| L0 | System Context | 注入身份、强制规则、协作约束 | `SOUL.md`、`USER.md`、`.alter0/agents/<agent_id>/AGENTS.md`、Agent Profile | 人工维护、控制面配置 | 每次请求固定注入 |
| L1 | Session History | 保留当前会话真实交互轨迹 | 会话消息主表、流式事件、工具轨迹 | 请求追加写入 | 最近消息窗口 |
| L2 | Session Summary | 承接长会话压缩结果与待办状态 | 会话摘要记录 | 系统自动压缩回写 | 每次请求按预算注入 |
| L3 | Daily Memory | 沉淀当天事实、偏好候选、任务结果候选 | `.alter0/memory/YYYY-MM-DD.md` | 系统抽取写入 | 按时间和相关性召回 |
| L4 | Long-term Memory | 沉淀跨天稳定事实、长期偏好、长期约束 | `.alter0/memory/long-term/MEMORY.md` | 系统升级写入或人工维护 | 按相关性与作用域召回 |
| L5 | Task Memory | 记录任务摘要、产物与回链关系 | `.alter0/tasks/*`、任务摘要索引 | 任务完成后系统写入 | 按 `task_id`、主题、时间召回 |

## 6. 作用域模型

每条记忆最小必须具备以下作用域之一：

1. `session`：仅当前 `session_id` 可见。
2. `principal`：当前用户主体可见；单用户部署默认取 `default`。
3. `agent_profile`：仅指定 `agent_id` 可见，例如 `.alter0/agents/<agent_id>/AGENTS.md`。
4. `workspace`：仅当前工作区或项目上下文可见。
5. `global`：用于 `SOUL.md` 等高优先级强制上下文。

默认隔离规则：

1. `Chat` 不继承其他 `Agent Profile` 私有记忆。
2. `Agent` 不默认继承其他 `agent_id` 的执行记忆。
3. `workspace` 级记忆只有在当前请求显式绑定工作区时才允许注入。
4. `global` 内容可覆盖普通长期记忆，但不得覆盖平台安全边界。

## 7. System Context 要求

### 7.1 文件优先级

| 内容 | 优先级 | 适用入口 |
| --- | --- | --- |
| 平台保留安全边界 | 最高 | Chat、Agent |
| `SOUL.md` | 最高 | Chat、Agent |
| `USER.md` | 高 | Chat、Agent |
| 当前 Agent 的 `.alter0/agents/<agent_id>/AGENTS.md` | 高 | Agent |
| Agent Profile | 高 | Agent |
| 运行时选择项（Provider / Model / Tools / Skills / MCP） | 高 | Chat、Agent |

### 7.2 加载顺序

上下文构建时，`system` 层按以下顺序加载：

1. 平台保留安全边界与运行时提示。
2. `SOUL.md`
3. `USER.md`
4. 当前 Agent 的 `.alter0/agents/<agent_id>/AGENTS.md`
5. Agent Profile 专属上下文
6. 本次请求的运行时选择项

### 7.3 基本要求

1. `SOUL.md` 冲突时优先于普通长期记忆、天级记忆与任务摘要。
2. `Chat` 默认不注入 Agent Profile。
3. `Agent` 每次执行必须注入当前 `agent_id` 对应 Profile。
4. `AGENTS.md` 不跨 Agent 共享；不同 Agent 只能读取和维护各自 `.alter0/agents/<agent_id>/AGENTS.md`。
5. `system` 文件变更后立即影响新请求，不依赖记忆索引刷新。

## 8. 会话历史与压缩要求

### 8.1 会话历史

1. `Chat` 与 `Agent` 请求统一落到会话持久化层。
2. 会话历史至少包含：`message_id`、`session_id`、`role`、`content`、`created_at`、`route`、`source`。
3. `Agent` 额外记录动作、观察、工具调用与完成结果，但默认不把完整原始日志回注到后续上下文。
4. 会话历史作为真实轨迹保留，不因 prompt 裁剪而被物理删除。

### 8.2 会话摘要

1. 长会话必须支持自动压缩，并将结果写为结构化摘要。
2. 摘要至少包含：当前目标、已确认事实、未完成事项、关键引用、最近偏好变更。
3. 摘要必须保留来源消息引用，支持回查原始消息。
4. 摘要与原始消息冲突时，以原始消息为准，并触发摘要重建。

### 8.3 请求级裁剪

1. 单次请求构建上下文时允许裁剪旧工具结果与重复观察文本。
2. 裁剪只影响当前 prompt，不改变持久化历史。
3. `Chat` 优先保留用户与助手自然语言消息。
4. `Agent` 优先保留最近目标、计划、关键观察与最新完成状态。

## 9. 天级记忆与长期记忆要求

### 9.1 Daily Memory

1. 每日一份 Markdown 文件，按日期命名。
2. 写入内容包括：用户偏好候选、任务摘要候选、重要决策候选、错误教训候选。
3. 同日允许多次追加，要求 append-only 或稳定合并。
4. 每条记录必须回链 `session_id`、`message_id`、`task_id` 中至少一项。

### 9.2 Long-term Memory

1. 使用 Markdown 作为主载体。
2. 仅允许写入稳定事实、长期偏好、长期约束、项目长期决策、已验证工作方式。
3. 支持从 Daily Memory 升级进入长期记忆。
4. 支持人工维护与系统自动升级共存。
5. 写入必须去重，避免同义事实重复堆积。

### 9.3 升级规则

满足以下条件之一的内容可升级为长期记忆：

1. 用户明确声明“以后都按这个规则执行”。
2. 同一偏好或事实在多个会话中重复出现且未被否定。
3. `Agent` 任务形成稳定项目级决策，并被后续任务继续引用。
4. 任务总结中明确标记为长期有效结论。

以下内容不得直接升级为长期记忆：

1. 单次任务的临时路径和调试细节。
2. 未确认正确性的推断。
3. 一次性产物链接或过期资源地址。
4. 与 `SOUL.md`、`USER.md`、当前 Agent 的 `AGENTS.md` 冲突的低优先级内容。

## 10. Task Memory 与 `.alter0` 运行态规范

### 10.1 运行态目录

所有任务历史数据默认写入 `.alter0/`。

```text
.alter0/
  tasks/
    index.json
    {task_id}/
      meta.json
      logs.jsonl
      artifacts.json
  memory/
    YYYY-MM-DD.md
    long-term/
      MEMORY.md
      YYYY-MM-DD.md
```

### 10.2 文件职责

1. `tasks/index.json`：任务全局索引，用于检索、分页和摘要聚合。
2. `tasks/{task_id}/meta.json`：任务状态机快照与基础字段。
3. `tasks/{task_id}/logs.jsonl`：阶段日志，append-only，支持流式追踪。
4. `tasks/{task_id}/artifacts.json`：产物列表与引用地址。
5. `memory/YYYY-MM-DD.md`：当日任务摘要与记忆候选。
6. `memory/long-term/YYYY-MM-DD.md`：长期候选摘要归档。

### 10.3 数据字段最小集

`Task Meta` 最小字段：

1. `task_id`
2. `session_id`
3. `source_message_id`
4. `task_type`
5. `status`
6. `created_at`
7. `finished_at`
8. `retry_count`

`Task Summary` 最小字段：

1. `task_id`
2. `goal`
3. `result`
4. `status`
5. `finished_at`
6. `tags`

### 10.4 回链与一致性

1. 摘要记录必须包含 `task_id`，支持从 Memory 视图回查详情。
2. 任务详情需展示其对应摘要引用，支持反向定位。
3. 摘要与任务主数据冲突时，以 `meta.json` 与 `logs.jsonl` 为准，并触发摘要重建。

### 10.5 留存策略

1. `logs.jsonl` 默认保留 90 天，可配置。
2. `index.json` 与 `meta.json` 保留全生命周期索引。
3. 天级摘要可长期保留；归档后仍保留 `task_id` 映射。
4. `.alter0/` 运行态目录默认由 `.gitignore` 忽略，不参与常规代码提交。

## 11. Chat 模块记忆需求

### 11.1 定位

`Chat` 以提升连续对话质量为目标，不默认进入强执行模式。

### 11.2 读取策略

1. 默认上下文由 `system` 文件、会话摘要、最近消息窗口组成。
2. 当问题涉及偏好、历史承诺、过往任务、历史结论时，允许触发长期记忆与任务摘要召回。
3. `Chat` 默认保守注入长期记忆，低相关结果不得进入主 prompt。
4. `Chat` 默认只注入少量高置信片段，避免历史噪声压制当前问题。

### 11.3 写入策略

以下场景应写入 Daily Memory 候选：

1. 用户明确给出稳定偏好、身份信息、长期目标。
2. 会话中形成后续仍需复用的明确约定。
3. 用户对既有记忆进行了纠正或覆盖。

以下场景不得自动写入长期记忆：

1. 即时情绪表达或一次性需求。
2. 模型推断出的偏好但用户未确认。
3. 仅在当前会话有效的临时上下文。

### 11.4 前端要求

1. 消息区应可标识“本次回复使用了记忆召回”。
2. 被召回记忆应展示来源类型，如长期记忆、天级记忆、任务摘要。
3. 对用户显式发出的“记住这件事”类请求，前端需给出保存结果反馈。
4. `Chat` 不新增独立 Memory 管理页，采用轻量提示、来源卡片与快捷操作。

## 12. Agent 模块记忆需求

### 12.1 定位

`Agent` 以提高执行连续性、沉淀可复用工作方式、减少重复探索为目标。

### 12.2 读取策略

1. `Agent` 每次执行必须加载 Agent Profile、会话摘要、最近执行轨迹。
2. 当任务与当前仓库、历史任务、长期规则相关时，优先召回长期记忆、天级记忆与任务摘要。
3. `Agent` 允许比 `Chat` 使用更高比例的任务记忆和项目决策记忆。
4. 执行过程中允许多次读取记忆，但每轮读取都必须受预算限制。

### 12.3 写入策略

以下场景必须写入 Daily Memory 或 Task Memory：

1. 任务完成后生成目标、结果、关键改动、产物、阻塞点摘要。
2. 执行中发现可复用规则、环境约束或项目决策。
3. 任务失败但形成明确教训、规避方式或重试前提。

以下场景应升级为长期记忆：

1. 多次执行验证过的仓库级规则。
2. 被 Agent Profile 或用户明确声明为长期有效的执行偏好。
3. 对后续任务稳定生效的环境约束与接入方式。

### 12.4 前端要求

1. `Agent -> Memory` 作为统一查阅入口，展示长期记忆、天级记忆、强制上下文、任务历史和本说明文档。
2. `Agent` 执行详情中需标记记忆读取与写入事件。
3. `Agent` 页面需支持从任务结果跳转到对应记忆或摘要来源。
4. Memory 页面默认只读；涉及编辑时使用显式操作，不在执行结果流中隐式改写。

## 13. 统一读取链路要求

一次请求的上下文构建顺序应遵循以下逻辑：

1. 注入平台保留 `system` 提示。
2. 注入 `SOUL.md`、`USER.md`、当前 Agent 的 `.alter0/agents/<agent_id>/AGENTS.md` 与 Agent Profile。
3. 注入当前会话摘要。
4. 注入最近消息窗口。
5. 根据请求主题召回长期记忆、天级记忆与任务摘要。
6. 根据预算裁剪、去重与排序。
7. 生成最终 prompt 并进入执行器。

排序优先级要求：

1. 与当前请求显式相关的记忆优先于时间更早但泛化的记忆。
2. 被用户明确确认的记忆优先于模型抽取的候选记忆。
3. 当前会话内新鲜事实优先于跨会话旧事实。
4. `SOUL.md` 与 `USER.md` 冲突时，以 `SOUL.md` 为准。

## 14. 统一写入链路要求

一次请求完成后，Memory 写入链路至少包含以下阶段：

1. 抽取：识别候选事实、偏好、决策、任务结果。
2. 分类：判定属于 `session`、`daily`、`long-term` 或 `task`。
3. 去重：与已有长期记忆和当日记忆进行语义及规则级去重。
4. 回链：记录 `session_id`、`message_id`、`task_id` 等来源。
5. 落盘：写入 Markdown 或任务索引。
6. 索引刷新：刷新派生检索索引。

写入链路必须满足以下要求：

1. 写入失败不影响主请求成功返回，但必须记录告警与重试状态。
2. 同一条事实重复抽取时，优先合并而非重复追加。
3. 自动写入的长期记忆应保留“来源于系统抽取”的标记。
4. 支持后台重建索引，不要求阻塞主线程。

## 15. 存储与索引要求

### 15.1 主存要求

1. 长期记忆与天级记忆统一使用 Markdown 存储。
2. 任务详情与原始日志继续使用 `.alter0/tasks/*` 结构。
3. 会话主数据继续沿用现有 Session 存储模型。
4. `SOUL.md`、`USER.md` 作为共享上下文文件单独维护；`AGENTS.md` 按 Agent 分散维护在 `.alter0/agents/<agent_id>/AGENTS.md`。

### 15.2 派生索引要求

1. 允许为 Markdown 记忆与任务摘要建立派生检索索引。
2. 派生索引不是事实源，损坏后必须可由 Markdown 主存与任务主数据重建。
3. 索引至少支持关键词检索、时间过滤、作用域过滤与相关性排序。
4. 后续可扩展向量召回，但不得替代 Markdown 主存方案。

## 16. Memory 接口要求

最小接口集合如下：

| 接口 | 说明 |
| --- | --- |
| `GET /api/memory/context-files` | 获取 `SOUL.md`、`USER.md`、Agent 私有 `AGENTS.md`、`MEMORY.md` 等文件列表与元信息 |
| `GET /api/memory/context-files/{id}` | 获取指定上下文文件内容 |
| `GET /api/memory/daily` | 获取天级记忆列表 |
| `GET /api/memory/daily/{date}` | 获取指定日期的天级记忆 |
| `GET /api/memory/long-term` | 获取长期记忆主文档与分段信息 |
| `GET /api/memory/search` | 按关键词、主题、时间与作用域搜索记忆 |
| `POST /api/memory/flush` | 触发当前会话或任务的记忆抽取与写入 |
| `GET /api/memory/tasks` | 获取任务摘要列表 |
| `GET /api/memory/tasks/{task_id}` | 获取任务详情回链信息 |
| `GET /api/memory/tasks/{task_id}/logs` | 获取任务日志游标读取结果 |
| `GET /api/memory/tasks/{task_id}/artifacts` | 获取任务产物列表 |
| `POST /api/memory/tasks/{task_id}/rebuild-summary` | 触发任务摘要重建 |

统一要求：

1. 查询接口默认只读。
2. 写接口必须带明确来源，如 `session_id`、`task_id`、`trigger_type`。
3. 搜索结果必须返回来源类型、作用域、时间与引用标识。

## 17. 前端 `Agent -> Memory` 页面要求

页面必须提供以下只读视图：

1. 长期记忆视图。
2. 天级记忆视图。
3. 强制上下文视图，如 `SOUL.md`。
4. 说明文档视图，即本文件全文。
5. 任务历史视图。

页面展示边界要求：

1. 说明文档视图需支持完整渲染本文件。
2. 支持按章节锚点快速定位。
3. 支持空态与加载失败提示。
4. 默认不提供在线编辑。
5. 任务历史视图默认展示摘要列表，详情按需加载。
6. 任务详情需展示基础信息、阶段日志与产物引用。
7. 支持从任务摘要跳转到对应会话，从会话回查对应任务。

## 19. 验收口径

1. 仓库内仅保留本文件作为 Memory 主说明文档，并可被稳定渲染。
2. `Chat` 能在同一会话内复用最近上下文，并在跨会话场景下召回高置信长期记忆。
3. `Agent` 能在执行任务时召回任务摘要、长期规则和项目决策，并在任务完成后写入摘要。
4. `SOUL.md`、`USER.md`、Agent 私有 `AGENTS.md`、长期记忆、天级记忆、任务历史的职责边界清晰且可视化可查。
5. Memory 检索结果可以回链到 `session_id`、`message_id` 或 `task_id`。
6. Markdown 主存与派生索引分离，索引异常时可重建。
7. `Chat` 与 `Agent` 共享底层 Memory 内核，但在注入预算、展示形式和写入策略上保持差异化。
